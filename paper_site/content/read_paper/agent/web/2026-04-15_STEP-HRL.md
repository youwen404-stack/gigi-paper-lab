# Hierarchical Reinforcement Learning with Augmented Step-Level Transitions for LLM Agents

- Category: agent
- Read date: 2026-04-15
- Paper date: 2026-04-07
- Authors: Shuai Zhen, Yanhua Yu, Ruopei Guo, Nan Cheng, Yang Deng
- Venue/Status: arXiv preprint, accepted to ACL 2026 Main Conference
- Source: https://arxiv.org/abs/2604.05808
- Code: https://github.com/TonyStark042/STEP-HRL

## Why this paper

这篇论文比较贴近当前 agent 方向里的一个核心痛点: 长程任务中的 LLM agent 往往越来越依赖超长历史上下文，导致推理成本上升、训练不稳定、泛化也受影响。它不是继续在“怎么压缩长上下文”上做修补，而是直接换了问题表述方式，这一点比较值得读。

## Core problem

作者关注的是交互式长程决策任务中的 LLM agent 训练问题。现有 RL-based agent 通常把策略建立在完整历史轨迹之上，随着任务推进，输入上下文不断变长，带来三个问题:

1. 推理 token 成本持续上升。
2. 冗余历史会淹没当前决策最关键的信息。
3. 高层规划和低层执行都被长上下文绑住，扩展性较差。

## Main idea

论文提出 STEP-HRL。核心思想是把 agent 的决策重构为“step-level transition”而不是“history-conditioned transition”。

方法上有两个关键结构:

1. Global progress
已完成的子任务序列被当作全局进度表示，高层策略据此决定下一个子任务。

2. Local progress
在每个子任务内部，再引入一个 local progress module，逐步把该子任务相关的动作和观察压缩成一段紧凑文本摘要。低层策略只看“当前子任务 + 当前观察 + local progress”，而不再看完整局部历史。

这样，高层与低层策略都不需要直接消费越来越长的原始轨迹。

## Training pipeline

作者采用两阶段训练:

1. Behavior Cloning
利用专家轨迹初始化高层策略、低层策略和 local progress policy。

2. Step-level Offline RL
在 step-level 状态定义上做 offline RL。三类策略共享同一个 policy backbone，但各自有单独 critic，从而兼顾参数效率和分层价值监督。

offline RL 部分结合了 implicit value learning 和 advantage-weighted regression，用 value 引导策略向更优动作分布靠近。

## Experimental setup

- Benchmarks: ScienceWorld, ALFWorld
- Backbones: Mistral-7B, Gemma-7B, Llama-3-8B
- Baselines: ReAct, Reflexion, SwiftSage, ETO, WKM, GLIDER

## Key results

作者报告 STEP-HRL 在两个 benchmark 上都明显优于已有方法。

- 在 ScienceWorld 上，相比强基线有稳定提升，seen/unseen 任务上都更强。
- 在 ALFWorld 上，成功率接近饱和，多个 backbone 下都超过 96%。
- 小模型也能受益，说明方法不只靠模型规模堆出来。
- 效率分析显示它的单步 token 使用更平稳，平均成本更低，说明 step-level formulation 不是只提升结果，也改善了推理成本结构。

## What feels important

我认为这篇论文最重要的不是“又做了一个 HRL agent”，而是它把 agent state representation 这个问题重新定义了:

- 不是默认认为长程任务就必须长上下文。
- 不是把历史原样塞进模型，而是把“进度”做成显式状态。
- 这让层级结构、状态压缩和 RL 优化真正对齐到同一个表示框架里。

## Quick takeaway

一句话总结: STEP-HRL 试图让 LLM agent 从“依赖整段历史做决策”转向“依赖结构化进度做决策”，因此同时改善了性能、泛化和 token 效率。
